# Deterministic Gate Boundaries Design

## Problem

`new_review_state()` now rejects a structurally valid bundle whose cached gate
does not match deterministic evaluation. Persistence and export boundaries still
use only `ReviewState.model_validate()` or `ReviewBundle.model_validate()`.
Pydantic proves structure and cross-references, but it does not recompute gate
truth.

A lifecycle state can therefore be mutated from a deterministic `blocked` verdict
to `ready`, saved, loaded, and rendered as Ready while `final_acceptance=False`.
Historical analysis bundles can carry the same contradiction in persisted or JSON
exported audit history.

## Approaches considered

1. **Shared core gate-boundary validators (selected).** Add a small module under
   `scopeproof_core/gates/` that reconstructs models through Pydantic and then
   compares every cached gate with `evaluate_gate()`. Lifecycle, storage, and
   reporting reuse it.
2. Duplicate the deterministic comparison in lifecycle, storage, and reporting.
   This invites message and coverage drift when gate rules change.
3. Put gate evaluation in Pydantic schema validators. The evaluator already
   depends on schema models, so this creates an undesirable dependency cycle or
   duplicates the truth table inside schemas.

## Design

Create two functions:

- `validated_review_bundle(bundle)` returns a reconstructed bundle only when its
  complete `GateDecision` equals fresh deterministic evaluation.
- `validated_review_state(state)` reconstructs the complete state and verifies
  the active bundle plus every analysis-history bundle.

Mismatches are rejected, never overwritten. Active/direct bundles retain the
stable error `analysis bundle gate must match deterministic evaluation`.
Historical mismatches use `analysis history bundle gate must match deterministic
evaluation` so operators can locate the invalid record.

Use these functions at:

- lifecycle transition input and initial-state creation;
- JSON save and load;
- every bundle or state export path.

The deterministic gate evaluator, schemas, gate semantics, file format version,
and rendered reports remain unchanged. No migration is manufactured for invalid
records: a contradictory cached gate must be regenerated by a trusted producer.

## Verification

Regression-first coverage must prove:

- a forged active Ready gate is rejected on save before any file is written;
- a forged persisted Ready gate is rejected on load;
- direct bundle and state exports reject forged Ready gates;
- a forged historical gate is rejected even when the active bundle is valid;
- valid lifecycle, storage round-trip, history, and exports remain unchanged.
